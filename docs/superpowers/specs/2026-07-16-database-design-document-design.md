# 数据库设计文档设计规格

日期：2026-07-16

状态：已实现

## 1. 背景与目标

江湖图谱已经形成 SQLite 与 Neo4j 并存的持久化架构，但仓库目前缺少一份统一、与代码实现同步的数据库设计文档。现有信息分散在 SQLAlchemy 模型、Neo4j Cypher、构建与审核规格、Docker Compose 配置和部署手册中，不利于新开发人员理解数据边界，也不利于部署人员执行备份、恢复、删除和故障排查。

本次工作新增 `docs/database-design.md`，同时服务两类读者：

1. 面向教学演示，解释为什么关系型数据库与图数据库并存，以及小说文本如何转化为本体实例、关系、属性和证据。
2. 面向开发与运维，给出可核对的物理表结构、图模型、索引约束、数据生命周期、备份恢复和典型查询。

文档必须以当前 `origin/master` 的代码为事实来源，不把历史规格中的计划字段误写成已实现结构。

## 2. 已批准的文档方案

采用“分层单文档”方案：正文先建立架构与数据流认知，再分别展开 SQLite 和 Neo4j 的逻辑及物理设计，最后通过附录提供完整字段字典、节点关系字典和查询示例。

选择单文档而不是拆分为开发版、运维版和教学版，原因如下：

- 双数据库之间存在明确映射，拆分后容易重复或产生版本偏差。
- 教学概念能够直接由真实物理结构支撑，避免形成只讲概念、不对应实现的材料。
- 运维操作需要同时理解 SQLite 控制面数据和 Neo4j 图谱数据，统一文档更适合执行备份与恢复。

## 3. 范围

### 3.1 本期包含

- SQLite 与 Neo4j 的职责边界和部署拓扑。
- 文本上传、任务处理、知识抽取、验证、图谱导入和审核的数据流。
- SQLite 中项目、任务、质量、审核、管理员、会话、登录限流和审计表。
- Neo4j 中项目、章节、实体、事实、证据和属性断言节点。
- Neo4j 中章节归属、实体归属、事实端点、实体关系、属性与证据关系。
- 主键、外键、唯一约束、索引、JSON 字段和状态字段的说明。
- `project_id` 项目隔离规则及 SQLite 项目与 Neo4j 项目的映射。
- 项目删除、完整构建、属性补抽、实体合并和审核操作对数据的影响。
- Docker 命名卷、备份、恢复、重建和一致性检查建议。
- 安全与隐私设计，包括密码摘要、会话令牌摘要、审计元数据和原文存储。
- 可直接用于排查的 SQL、Cypher 和容器命令示例。
- 当前设计限制和后续演进建议。

### 3.2 本期不包含

- 修改任何数据库模型、API、业务逻辑或部署配置。
- 引入数据库迁移框架或自动生成 DDL。
- 将 Neo4j 模型改写为 RDF、OWL 三元组存储或其他图数据库实现。
- 为尚未实现的字段、索引或关系给出“当前已存在”的表述。
- 记录真实生产密码、Azure OpenAI Key、Cookie、会话令牌或小说全文。

## 4. 事实来源与核对规则

正文编写按以下优先级核对事实：

1. `apps/api/src/app/**/models.py`：SQLite 字段、类型、默认值、外键和唯一约束。
2. `apps/api/src/app/graph/neo4j.py`：Neo4j 约束、索引、节点与关系写入结构。
3. 各模块 `repository.py`、`service.py` 和 `review/graph.py`：数据生命周期、事务、删除、实体合并与兼容升级行为。
4. `apps/api/src/app/settings.py` 与 `compose.yaml`：连接配置、数据库文件位置和 Docker 数据卷。
5. 自动测试：验证接口与数据操作的预期行为。
6. 历史规格和部署手册：只用于补充背景，不覆盖当前代码事实。

每个物理结构章节都要标明数据库类型，并区分三种状态：

- 已实现：能在当前代码中找到直接证据。
- 运行时生成：由 SQLAlchemy `create_all` 或启动兼容升级生成。
- 建议：属于运维或后续演进建议，不得与已实现结构混写。

## 6. 图示设计

文档使用 Mermaid，确保 GitHub 可直接渲染，并避免依赖外部图片。

### 6.1 总体架构图

展示浏览器、Web、API、Worker、SQLite、上传文件目录和 Neo4j 的连接方向，强调：

- SQLite 是控制面和工作流存储。
- Neo4j 是知识图谱查询存储。
- 原始 TXT 位于应用数据卷，不直接存入 SQLite 或 Neo4j 全文节点。
- API 与 Worker 共享 SQLite 和上传目录，API/Worker 按职责访问 Neo4j。

### 6.2 SQLite ER 图

ER 图覆盖：

- `projects` 到 `jobs` 的一对多关系。
- `jobs` 到 `job_events` 的一对多关系。
- `jobs` 到 `job_quality` 的一对一关系。
- 管理员账号到会话的一对多关系。
- 审核表按 `project_id` 和业务标识关联，但当前未声明数据库外键的事实。

### 6.3 Neo4j 图模型图

图模型至少展示：

```text
(:Project)-[:HAS_CHAPTER]->(:Chapter)
(:Project)-[:HAS_ENTITY]->(:Entity)
(:Fact)-[:SOURCE]->(:Entity)
(:Fact)-[:TARGET]->(:Entity)
(:Entity)-[:RELATED]->(:Entity)
(:Fact)-[:EVIDENCED_BY]->(:Evidence)-[:IN_CHAPTER]->(:Chapter)
(:Entity)-[:HAS_ATTRIBUTE]->(:AttributeAssertion)-[:EVIDENCED_BY]->(:Evidence)
```

同时解释 `Fact` 节点与 `RELATED` 关系并存的原因：前者承载可审核、可追溯的关系事实和证据，后者服务邻居遍历与图谱展示。

## 7. 字典表达规范

### 7.1 SQLite 表字典

每张表按统一格式记录：

- 用途。
- 主键。
- 字段名、SQLAlchemy 类型、空值规则、默认值和说明。
- 外键及删除行为。
- 唯一约束和索引。
- 主要写入方与读取方。
- 生命周期和清理规则。

对于 JSON 字段，应列出当前实际载荷结构示例，而不是只写“JSON 数据”。

### 7.2 Neo4j 字典

每类节点记录：

- 标签和业务语义。
- 业务键及唯一约束。
- 主要属性。
- 入边与出边。
- 创建、更新和删除方式。

每类关系记录：

- 起点、终点和方向。
- 关系属性。
- 是否为查询加速投影，还是领域事实本身。
- 对应证据和审核方式。

## 8. 一致性与生命周期说明

文档需要明确当前系统不是跨数据库分布式事务，并说明实际一致性边界：

- SQLite 先记录项目和任务状态，Worker 再按阶段向 Neo4j 幂等写入。
- Neo4j 使用 `MERGE`、复合唯一约束和稳定 ID 支持重试。
- 失败任务通过状态和错误码暴露，不能把“SQLite 中任务完成”与“Neo4j 一定完整”简单等同。
- 项目删除需要同时处理 Neo4j 中带 `project_id` 的节点、Neo4j `Project` 节点、SQLite 项目及其级联任务数据、上传目录原文。
- 审核项和审核动作当前通过业务字段关联项目和图事实；没有数据库外键时应显式说明清理责任。
- 实体合并会迁移图关系、事实端点和别名；当前未迁移属性归属，正文必须将其列为已知限制。

## 9. 运维章节要求

备份与恢复以 Docker Compose 默认部署为基准，覆盖：

- `app-data`：SQLite 数据库及上传文本。
- `neo4j-data`：Neo4j 数据目录。
- `.env` 或部署环境变量：只说明需要备份配置，不在示例中输出密钥。

备份流程必须说明一致性窗口。推荐停写或短暂停止 API/Worker 后，对两个数据卷在同一维护窗口执行备份；如果在线备份，应明确可能出现 SQLite 与 Neo4j 时间点不一致。

恢复流程必须包括：

1. 恢复配置和数据卷。
2. 启动 Neo4j、API、Worker 和 Web。
3. 检查健康状态。
4. 比对项目数、任务状态和各项目图节点数。
5. 对未完成任务执行人工判断或重新构建。

破坏性命令必须配有风险提示，不把删除卷命令作为普通重启步骤。

## 10. 安全与隐私要求

- 管理员密码只保存 Argon2id 摘要，不展示摘要示例。
- 会话只保存令牌摘要，原始令牌通过 HttpOnly Cookie 传输。
- 审计元数据必须排除密码、Cookie、会话令牌、CSRF 和模型 API Key。
- 数据库设计文档不得复制真实生产连接密码或 Azure OpenAI Key。
- 小说原文和证据摘录属于受保护内容，备份、日志和导出需要按部署方数据策略管理。
- Neo4j 浏览器端口和 Bolt 端口的公开暴露风险应在部署建议中说明。

## 11. 验收标准

文档完成时必须满足：

1. `docs/database-design.md` 存在，目录完整且 GitHub Markdown 可读。
2. SQLite 表清单与当前 `origin/master` 中 SQLAlchemy 模型一致。
3. Neo4j 节点、关系、唯一约束和索引与 `graph/neo4j.py` 一致。
4. 至少包含总体架构图、SQLite ER 图和 Neo4j 图模型图。
5. 明确区分已实现结构与建议项。
6. 覆盖 Docker 数据卷、备份恢复和一致性检查。
7. SQL 与 Cypher 示例不包含真实凭据，并使用参数或占位符表达项目 ID。
8. 运行 Markdown 结构检查，不出现失效的仓库内链接或未闭合代码块。
9. 执行 `git diff --check`，无空白错误。
