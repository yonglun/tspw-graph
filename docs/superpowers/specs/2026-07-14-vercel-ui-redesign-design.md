# 江湖图谱 Vercel 风格前端重构设计

## 背景

现有前端在 `theme.css` 上叠加 `swiss.css`，形成红色、荧光黄、粗边框、装饰网格和 700/800 字重主导的视觉体系。该方案与本次指定的 Vercel 设计语言冲突，也让图谱、问答、审核和构建页面之间出现不同程度的视觉噪声与密度不一致。

本次重构在不改变业务流程、API、路由和数据结构的前提下，建立统一的 Vercel 风格设计系统。界面应呈现克制、精确、工程化的体验，优先保证内容可扫描、操作可发现、状态可理解。

## 目标

- 全站采用灰阶界面，蓝色只用于交互、选中与键盘焦点。
- 用 shadow-as-border 统一容器边界，移除装饰性实线边框。
- 建立 Geist 风格的字体、字重、字距、圆角、间距与阴影令牌。
- 统一导览、本体、图谱、故事线、问答、构建和审核页面的信息层级。
- 保留实体类型颜色，但只用于 8–10px 数据指示点。
- 支持 320、768、1024、1440px 关键视口。
- 满足 WCAG 2.1 AA 的键盘焦点、语义结构与状态表达要求。

## 非目标

- 不修改后端 API、数据库结构或知识抽取逻辑。
- 不重写路由、项目选择状态或数据获取层。
- 不引入重量级组件库或全局状态管理方案。
- 不为视觉效果增加装饰性动画、渐变、纹理或背景图案。
- 不扩大为通用组件库项目；只抽取当前页面实际共享的设计基础。

## 实施策略

采用集中式设计系统层替换现有 `swiss.css`。保留现有 React 页面结构和业务组件，仅对影响信息层级、可访问性或响应式行为的组件结构做小幅调整。

相比逐页重写，该策略能减少重复规则并降低回归风险；相比先建设完整组件库，该策略更符合当前项目规模，也能在一个增量中完成全站一致性改造。

## 设计令牌

### 色彩

| 角色 | 值 | 用途 |
| --- | --- | --- |
| 页面背景 | `#FAFAFA` | 页面画布 |
| 提升表面 | `#FFFFFF` | 卡片、面板、菜单 |
| 凹陷表面 | `#F2F2F2` | 输入辅助区、次级控件 |
| 悬停表面 | `#EBEBEB` | 导航与 ghost 按钮悬停 |
| 主文字 | `#171717` | 标题、主要内容 |
| 次文字 | `#4D4D4D` | 导航、说明、辅助内容 |
| 弱文字 | `#8F8F8F` | 占位符、禁用状态 |
| 交互蓝 | `#0072F5` | 链接、选中、外层焦点环 |
| 输入焦点蓝 | `#005FCC` | 输入框原生焦点 |

品牌标识改为纯黑白。红、黄等色彩不得作为装饰或大面积背景。实体类型与业务状态可使用既有类别颜色，但必须限制在 8–10px 圆点，并同时显示文字标签，避免只依赖颜色传达信息。

### 字体

- 界面字体：`Geist Sans`，回退到 `Inter`、`Helvetica Neue`、`Arial`、`PingFang SC` 和系统无衬线。
- 代码字体：`Geist Mono`，回退到 `ui-monospace`、`SFMono-Regular`、`Roboto Mono`。
- 只使用 400、500、600 三种字重。
- 页面主标题：48px / 48px / 600，字距 `-2.28px`。
- 区块标题：32px / 40px / 600，字距 `-1.28px`。
- 小标题与标签：14px / 20px / 500。
- 正文：16px / 24px / 400。
- 辅助文字：12px / 16px / 400。
- 代码：13px / 20px / 500。

### 间距与尺寸

所有间距使用 4px 基准令牌：4、8、12、16、24、32、40、48、64、96、128、192、256px。

- 标准内容宽度：1200px。
- 工作台最大宽度：1400px。
- 页面水平边距：24px；移动端为 16px。
- 全局导航高度：64px。
- 默认控件高度：40px；紧凑控件为 32px；大控件为 48px。

### 圆角与深度

- 默认控件圆角：6px。
- 卡片和浮层圆角：12px。
- 标签与状态徽标：9999px。
- 容器边界：`0 0 0 1px rgba(0, 0, 0, 0.08)`。
- 小型提升：边界阴影加 `0 2px 2px rgba(0, 0, 0, 0.04)`。
- 菜单和浮层按层级叠加 4–24px 的低透明度阴影。
- 容器不使用 CSS `border` 作为视觉边界。

### 焦点与运动

- 按钮、链接和复合控件使用双环焦点：`0 0 0 2px #FFF, 0 0 0 4px #0072F5`。
- 输入框使用 `#005FCC` 的可见原生轮廓。
- hover 只改变背景色和文字色，不使用缩放、位移或透明度动画。
- 菜单和浮层最多使用 200ms 的轻微出现动效。
- 当系统启用 `prefers-reduced-motion` 时关闭非必要动效。

## 全局结构

### 导航

全局导航固定为 64px 高，背景为 `#FAFAFA`，底部分隔使用阴影而非实线。内容最大宽度为 1400px。

- 左侧：纯黑白“江湖图谱”品牌标识。
- 中间：导览、本体、图谱、故事线、问答、构建、审核。
- 右侧：当前项目选择器。
- 当前导航项使用浅灰背景和主文字；不使用彩色下划线。
- 窄屏下品牌与项目选择器保持可见，导航在下一行横向滚动。

### 页面容器

阅读型页面使用 1200px 最大宽度；图谱与审核等工作台使用 1400px。页面标题区遵循一致的标题、说明和操作布局，避免每个页面独立定义大幅海报式头部。

## 页面设计

### 导览

导览页改为内容优先的产品介绍布局。标题、说明、步骤和主要操作位于标准内容容器内。三元组示例放入白色提升卡片，使用紧凑节点与连接关系，不再使用黑色画布、网格或印章装饰。

### 本体

- TBox/ABox 使用标准分段控件。
- 类型卡片在大屏为三列、中屏两列、移动端一列。
- 类型颜色仅显示为小圆点。
- 属性在对应卡片内部展开，并保持文档流，不能覆盖或挤压相邻卡片。
- ABox 示例使用紧凑白色关系卡片。

### 图谱

- 页面标题压缩为工具栏，搜索框是主要入口。
- 画布使用纯白背景，不显示网格、渐变或纹理。
- 节点主体保持低饱和灰阶；实体类型用小圆点或节点内小标记表示。
- 选中节点和关系使用蓝色外环，并提供非颜色的选中状态文本。
- 搜索结果使用 menu 层级阴影和 12px 圆角。
- 实体详情为右侧浮动面板，内部按概述、属性、关系、属性证据、关系证据分组。
- 移动端详情面板置于画布下方。

### 故事线

时间线使用间距和轻量阴影分组，事件节点颜色限制为小状态点。章节、事件类型、标题和描述建立清晰的文本层级；空状态保留明确说明。

### 问答

- 大屏使用输入区与答案区双栏布局，移动端改为单栏。
- 示例问题采用 ghost 按钮。
- 答案卡显示结论、图路径、原文证据与折叠技术细节。
- 路径节点使用紧凑标签，关系文字使用次级色，不使用大面积彩色背景。
- 加载状态使用行级骨架或文字状态，错误显示在答案工作区并允许重试。

### 构建

保留“来源—流水线—质量”三阶段结构。三个区域使用一致的白色面板、40px 表单控件和紧凑状态展示。进度、错误与质量指标同时使用文字和小状态点表达。

### 审核

- 保持“审核队列—审核详情”的主从结构。
- 队列项突出可读三元组和审核原因，不优先展示技术 ID。
- 严重度使用小状态点加文字，不使用整块警示色背景。
- 详情依次展示原因、事实三元组、原文证据和审核动作。
- 技术标识默认折叠。
- 移动端先显示队列，选择后滚动到详情。

## 组件边界

本次保留现有页面组件与数据获取逻辑。新增或调整的共享样式职责如下：

- `App` / `SiteHeader`：全局导航、品牌、项目切换和移动端导航。
- 设计系统样式：颜色、字体、间距、圆角、阴影、焦点和响应式令牌。
- 页面级工作区：只处理页面布局，不重复定义基础控件视觉。
- 状态点：统一承载实体类别、任务状态和审核严重度；始终配合文字。

如结构调整导致单个组件超过 200 行，应拆分展示组件，但不引入新的全局状态层。

## 数据流与行为保持

- 项目选择继续由 `ProjectContext` 管理，并保留 URL 中的 `project` 参数。
- 图谱搜索、邻居加载、证据定位和审核跳转保持现有 API 调用。
- 问答仍使用现有模板、图路径和证据响应。
- 构建和审核操作保持原有提交、暂停、继续、取消、重试、接受、拒绝和忽略行为。
- 视觉状态不得改变服务端状态或数据缓存策略。

## 加载、错误与空状态

- 内容加载使用骨架屏或局部 `role="status"` 文本。
- API 错误显示在所属工作区内，使用文字、图标或状态点共同表达，并提供现有重试操作。
- 空状态必须说明当前为空的原因或下一步行动。
- 不允许因加载或错误让页面只剩空白。
- 状态消息应通过 `aria-live` 或合适的语义角色通知辅助技术。

## 响应式规则

### 320px

- 页面单栏，水平边距 16px。
- 导航横向滚动。
- 图谱详情置于画布下方。
- 表单按钮可占满宽度。

### 768px

- 本体卡片两列。
- 问答和审核切换为单栏或上下工作区。
- 工具栏允许换行，但搜索和主操作保持优先。

### 1024px

- 保留紧凑双栏布局。
- 缩小侧边面板和页面边距。
- 导航可横向滚动，避免压缩文字。

### 1440px 及以上

- 阅读内容限制为 1200px。
- 图谱和审核工作台可扩展到 1400px。
- 超宽屏不无限拉伸正文和证据内容。

## 可访问性

- 所有交互元素必须可通过键盘访问。
- 属性和关系行支持 Enter 与 Space，并优先使用原生按钮语义。
- 图标按钮必须有可读 `aria-label`。
- 分段控件使用可理解的选中状态。
- 所有文字与背景满足 WCAG 2.1 AA 对比度。
- 颜色不能成为状态或实体类型的唯一表达方式。
- 页面保持单一 `h1`，标题层级连续。
- 跳到主要内容入口在键盘聚焦时可见。

## 测试与验收

### 自动化验证

- 现有 Vitest 前端测试全部通过。
- TypeScript 类型检查通过。
- Vite 生产构建通过。
- 新增或调整结构时补充导航、键盘和状态渲染测试。

### 视觉与交互验证

- 在 320、768、1024、1440px 下检查无横向溢出、遮挡或文本重叠。
- 键盘遍历导航、项目选择、搜索、分段控件、属性、关系和审核动作。
- 检查加载、错误和空状态均可见且语义正确。
- 检查控制台无运行时错误和可访问性警告。

### 设计一致性验收

- 不存在红黄装饰、背景网格、装饰性渐变或大面积状态色。
- 交互蓝以外的颜色只出现在小型类别或状态指示点。
- 不存在 700 或 800 字重。
- 容器边界使用 shadow-as-border，不使用实线 CSS border。
- 间距均落在 4px 基准系统内。
- 功能、API、路由和项目保持行为与重构前一致。

## 风险与控制

- `theme.css` 仍包含旧视觉规则。实施时由新的设计系统层完整覆盖并逐步移除冲突规则，避免长期双重维护。
- 当前仓库未内置 Geist 字体资源。首个版本使用 Geist 优先的系统字体栈；如需完全一致，可后续在确认字体许可与资源策略后本地托管字体。
- 图谱节点本身需要类别辨识。类别色将保留为小尺度数据标记，同时通过文字图例保证可理解性。
- 全站 CSS 影响面较大。实施按基础令牌、全局导航、阅读页面、工作台页面的顺序增量验证。
