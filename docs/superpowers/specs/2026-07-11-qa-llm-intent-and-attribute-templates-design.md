# 问答 LLM 意图解析与属性问答模板设计

日期：2026-07-11

状态：已确认设计，待实施计划

## 背景与目标

当前问答服务已经支持少量关系模板，但问句解析规则有限。例如，图谱中存在 `令狐冲 - MEMBER_OF - 华山派`，而“令狐冲隶属于什么门派？”因主体提取失败无法命中实体。与此同时，实体属性已经入图并能在实体详情中显示，但问答服务尚未读取属性断言。

本次改造目标：

1. 新增属性问答模板，支持性别、身份、称号、人物状态等本体属性。
2. 优化关系问句解析，支持“属于哪个门派”“隶属于什么门派”等自然问法。
3. 使用 Azure OpenAI 作为本地规则无法识别时的意图解析兜底。
4. 保持答案可解释：LLM 只解析意图，后端只执行白名单查询，并且必须返回图谱证据。

## 非目标

- 不让 LLM 直接生成最终答案。
- 不让 LLM 直接生成或执行任意 Cypher。
- 不在本次改造中实现开放域知识问答。
- 不改变现有实体、关系、属性抽取和审核数据模型。

## 方案比较与选择

### 方案一：本地模板优先，LLM 兜底（采用）

常见问法走本地解析器；无法识别时调用 Azure OpenAI 输出受限 JSON 意图。该方案兼顾响应速度、额度消耗、可解释性和自然语言覆盖范围。

### 方案二：所有问题都先经过 LLM

实现统一，但每次问答都增加模型延迟和额度消耗，并扩大模型不可用时的影响面。

### 方案三：LLM 生成多步查询计划

支持复杂组合问题，但查询计划校验、安全边界和调试成本明显增加。本次只预留意图结构，不实现多步计划执行。

## 处理架构

问答链路如下：

1. 接收当前项目和用户问题。
2. 对问题做本地归一化：简繁体统一、去除礼貌前缀和疑问尾词、识别常见关系/属性关键词。
3. 若本地解析成功，生成受控意图。
4. 若本地解析失败，调用 Azure OpenAI 生成受限 JSON 意图。
5. 校验意图：实体唯一匹配、关系和属性白名单、置信度阈值。
6. 根据意图选择固定 Neo4j 查询模板。
7. 只在查询返回事实或属性证据时生成回答；否则明确拒答。
8. 返回答案、路径或属性值、原文证据和技术细节。

意图类型：

```text
RELATION       查询实体之间的关系
ATTRIBUTE      查询实体属性
INTRODUCTION   查询实体简介
UNSUPPORTED    无法安全映射的问题
```

## 问答模板

### 属性模板

| 问法关键词 | 本体属性 |
| --- | --- |
| 性别、男还是女 | `gender` |
| 身份、是什么身份、担任什么角色 | `identity` |
| 称号、尊称、外号 | `honorific` |
| 生死、是否在世 | `life_status` |
| 活动区域、所在地区 | `activity_region` / `region` |
| 特征、特点 | `characteristic` |

属性查询必须返回属性断言及其证据，不能仅返回没有来源的属性值。

### 关系模板

| 问法关键词 | 关系类型 |
| --- | --- |
| 师父、师傅、拜谁为师 | `MASTER_OF` |
| 属于、隶属、加入哪个门派 | `MEMBER_OF` |
| 掌握、会什么武功 | `KNOWS` |
| 敌人、对手 | `ENEMY_OF` |
| 盟友、朋友 | `ALLY_OF` |
| 持有、拥有 | `HOLDS` |
| 发生在哪里 | `LOCATED_IN` |

关系模板继续沿用现有关系方向和证据返回逻辑。

## 问句归一化与主体解析

本地解析器需要：

- 统一简体和繁体字符；
- 去除“请问、告诉我、能否告诉我”等礼貌前缀；
- 去除“是什么、有哪些、是谁、哪一个、哪个、什么”等疑问尾词；
- 识别“属于哪个门派”“隶属于什么门派”“拜入了哪个门派”等关系句式；
- 使用当前项目实体规范名和别名做匹配；
- 精确命中唯一实体时直接使用；
- 多个同名实体无法区分时拒答，不猜测。

示例：

```text
令狐冲隶属于什么门派？ -> subject=令狐冲, intent=RELATION, relation=MEMBER_OF
令狐冲属于哪个门派？   -> subject=令狐冲, intent=RELATION, relation=MEMBER_OF
令狐冲的性别是什么？   -> subject=令狐冲, intent=ATTRIBUTE, property=gender
```

## LLM 意图契约

LLM 输入包括用户问题、实体类型目录、关系白名单和属性白名单。LLM 只能输出如下结构：

```json
{
  "intent": "RELATION",
  "subject": "令狐冲",
  "relation": "MEMBER_OF",
  "property": null,
  "confidence": 0.96
}
```

约束：

- `RELATION` 必须提供白名单中的 `relation`；
- `ATTRIBUTE` 必须提供白名单中的 `property`；
- `subject` 必须来自用户问题中的实体名称或别名；
- 置信度低于阈值时拒答；
- JSON 无法解析、模型限流、额度不足或内容过滤时拒答；
- 禁止输出 Cypher、任意字段名或自由文本答案。

LLM 复用服务端已有 Azure OpenAI 配置，前端不接触 API Key。

## 错误处理与可观测性

以下情况统一返回明确拒答，不调用开放式生成：

- 无法识别意图；
- 实体不存在或匹配到多个实体；
- 关系/属性不在白名单；
- 没有对应事实或属性证据；
- LLM 返回非法 JSON、低置信度、限流或不可用。

记录以下指标：本地命中或 LLM 命中、LLM 耗时、解析失败原因、实体匹配结果、是否找到证据。日志中不得记录 API Key。

## 前端体验

问答页新增示例问题：

- 令狐冲的师父是谁？
- 令狐冲属于哪个门派？
- 令狐冲的性别是什么？
- 令狐冲有哪些称号？

回答卡片保持现有结构，展示答案、图谱路径或属性值、原文证据和技术细节。技术细节展示实际意图、查询模板和参数，不展示敏感配置。

## 测试与验收标准

### 后端测试

- 本地解析覆盖“属于哪个门派”“隶属于什么门派”等关系句式；
- 属性解析覆盖 `gender`、`identity`、`honorific`、`life_status`；
- LLM JSON 合法性、白名单和低置信度校验；
- 属性查询返回值和属性证据；
- 关系查询返回路径和关系证据；
- 无实体、歧义实体、无证据和模型不可用时拒答；
- 既有固定问题回归测试。

### 前端测试

- 示例问题可填入并提交；
- 属性答案展示属性值和证据；
- 关系答案展示路径和证据；
- 拒答状态和技术细节正确显示。

### 验收标准

1. “令狐冲隶属于什么门派？”返回“华山派”及原文证据。
2. “令狐冲的性别是什么？”返回“男”及属性证据。
3. 同义问法能够定位同一实体和关系。
4. 不支持的问题仍然拒答，不产生模型编造答案。
5. LLM 返回非法关系、非法属性或低置信度时不执行查询。
6. LLM 不可用时，既有固定问法仍然可用。
