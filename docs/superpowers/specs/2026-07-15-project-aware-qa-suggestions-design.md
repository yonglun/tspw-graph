# 项目感知的问答示例设计

## 背景

问答页当前把项目名称和“令狐冲”相关示例写死在前端。用户切换到《书剑恩仇录》等其他项目后，页面仍显示“向《笑傲江湖》图谱提问”，示例问题也仍属于《笑傲江湖》，造成项目上下文错乱，并可能引导用户提出当前图谱无法回答的问题。

## 目标

- 问答页标题始终使用当前项目的真实显示名称。
- 示例问题从当前项目的 Neo4j 数据动态生成，而不是绑定某部小说或固定人物。
- 只展示现有问答引擎能够回答、且当前图谱中有原文证据支持的问题。
- 项目切换时立即清除旧项目的答案与示例，不显示跨项目的短暂残留。
- 请求乱序返回时，旧项目响应不得覆盖新项目页面。

## 非目标

- 不使用 LLM 生成示例问题。
- 不扩大本次问答引擎支持的关系或属性范围。
- 不修改图谱抽取、实体消歧或证据审核规则。
- 不把管理员权限引入只读的问答示例接口。

## 用户体验

问答页以当前项目为唯一上下文来源：

1. 页面标签显示“向《{当前项目名称}》图谱提问”。
2. 页面进入或项目切换后，加载该项目的动态建议。
3. 后端选择一个信息最丰富且可回答问题最多的 `Person` 作为代表人物，并返回最多 6 个建议问题。
4. 用户点击建议后，问题填入输入框；提交仍沿用现有 `/api/ask` 流程。
5. 图谱为空、没有合格人物或没有可验证问题时，不回退到“令狐冲”或其他项目的固定示例，仅显示简短空状态：“当前项目暂无可推荐的问题”。用户仍可自行输入问题。
6. 加载失败时显示非阻塞提示，并保留手动提问能力。

项目切换时前端同步执行以下重置：清空输入框、上一次回答、技术详情展开状态和旧建议；随后请求新项目建议。这样不会让旧答案被误认为属于新项目。

## API 设计

新增公开只读接口：

```http
GET /api/projects/{project_id}/qa-suggestions
```

成功响应：

```json
{
  "project_id": "project-123",
  "project_title": "书剑恩仇录",
  "representative_entity": {
    "id": "project-123:entity:chen-jialuo",
    "name": "陈家洛",
    "type": "Person"
  },
  "suggestions": [
    {
      "id": "relation:MEMBER_OF",
      "question": "陈家洛属于哪个门派？",
      "kind": "relation",
      "capability": "MEMBER_OF"
    },
    {
      "id": "attribute:gender",
      "question": "陈家洛的性别是什么？",
      "kind": "attribute",
      "capability": "gender"
    }
  ]
}
```

如果项目存在但没有合格建议，返回 `200`，`representative_entity` 为 `null`，`suggestions` 为空数组；项目不存在返回现有风格的 `404 PROJECT_NOT_FOUND`。Neo4j 不可用时沿用图谱接口的 `503 GRAPH_UNAVAILABLE` 映射。

建议条目使用稳定的能力 ID，便于测试、埋点和未来扩展。响应不返回 Cypher、提示词或敏感配置。

## 后端设计

### 职责边界

- `projects` 层校验项目存在并提供项目标题。
- `qa` 层负责代表人物排序、可回答能力判定和问题文案生成。
- `graph` repository 新增一次有界、只读的聚合查询，返回候选人物及其可验证关系/属性能力。
- Router 组装响应并复用现有图谱异常到 HTTP 状态的映射。

不为每个人物逐个调用 `entity_detail`，避免 N+1 查询。Neo4j 查询只扫描指定 `project_id`，限制候选数和最终建议数。

### 代表人物选择

候选范围为当前项目中未被合并的 `Person`。只统计未被拒绝并且至少关联一条有效 Evidence 的事实或属性断言。

按以下顺序降序排序：

1. 可验证关系事实数量；
2. 可回答属性断言数量；
3. 不同可回答能力数量；
4. 规范名称、实体 ID 升序，作为稳定平局规则。

如果最高候选没有任何可回答能力，则不返回代表人物。

### 可回答问题模板

关系建议必须同时满足关系方向、实体角色和证据要求，并与现有 `RELATION_TEMPLATES` 保持一致：

| 能力 | 数据条件 | 建议文案 |
| --- | --- | --- |
| `MASTER_OF` | 人物是师承事实的目标（弟子） | `{人物}的师父是谁？` |
| `KNOWS` | 人物是武学掌握事实的主体 | `{人物}掌握什么武功？` |
| `MEMBER_OF` | 人物是组织隶属事实的主体 | `{人物}属于哪个门派？` |

属性建议仅在人物存在同名 `property_id`、非空值且至少有一条原文证据时生成：

| 属性 | 建议文案 |
| --- | --- |
| `gender` | `{人物}的性别是什么？` |
| `identity` | `{人物}是什么身份？` |
| `honorific` | `{人物}有哪些称号？` |
| `life_status` | `{人物}是否在世？` |
| `activity_region` | `{人物}的活动区域在哪里？` |
| `region` | `{人物}的所在区域是哪里？` |
| `characteristic` | `{人物}有什么特点？` |

属性清单与本地问句解析器 `_ATTRIBUTE_MARKERS` 的受控能力保持同步。未来新增能力时，应同时补充解析、回答、建议模板及测试，避免建议与实际回答能力漂移。

建议排序优先关系问题，再按 `MASTER_OF`、`MEMBER_OF`、`KNOWS`，随后按上述属性顺序；最多返回 6 条。每项能力只生成一个问题。

## 前端设计

`AskPage` 从 `ProjectContext` 获取当前 `projectId` 和项目标题，不再包含小说名或人物名常量。

新增 API client 类型和 `getQaSuggestions(projectId, signal)` 方法。页面在 `projectId` 变化时：

1. 同步重置问题、回答、错误、技术详情和建议列表；
2. 取消上一个项目的 `AbortController`；
3. 发起新请求并显示轻量加载占位；
4. 仅当响应仍对应当前 `projectId` 时写入状态；
5. 把服务端返回的 `project_title` 用于提问标签，若建议请求尚未完成则使用 `ProjectContext` 中已加载的标题。

建议按钮继续复用现有交互和可访问性样式。空数组时显示空状态，不渲染固定兜底问题。

## 错误与并发处理

- `AbortError` 不显示错误提示。
- 网络或 Neo4j 错误只影响建议区域，不禁用输入框和“查询图谱”按钮。
- 以请求对应的项目 ID 和取消信号双重防止旧响应覆盖新项目。
- 项目切换后旧答案必须立即消失，即使新建议仍在加载。
- API 返回未知能力时前端仍可显示问题文本；能力字段只作元数据，不参与问答提交。

## 性能与安全

- 建议查询为只读、单项目、有界查询，不调用 LLM，不产生模型费用。
- 项目 ID 必须作为所有 Neo4j 匹配条件，防止跨项目数据混合。
- 只统计未合并实体、未拒绝事实和带证据数据，保持与问答引擎的可信回答约束一致。
- 响应最多 6 条建议，候选人物聚合在数据库侧完成。
- 该接口与项目列表、图谱浏览一样保持公开只读；构建、审核和管理员管理的鉴权边界不变。

## 测试策略

### 后端

- 项目标题原样返回。
- 从不同项目的数据分别选择代表人物，结果不串项目。
- 排名优先选择关系/属性/证据更丰富的人物，平局结果稳定。
- 只为方向正确、未拒绝且有证据的关系生成建议。
- 只为非空且有证据的受支持属性生成建议。
- 最多返回 6 条且顺序稳定。
- 空图谱、无人物、无证据时返回空建议。
- 项目不存在返回 404，Neo4j 不可用返回 503。

### 前端

- 标签使用当前项目名称，不包含硬编码《笑傲江湖》。
- 切换项目后旧建议和旧答案立即清空。
- 新项目建议成功后显示对应人物问题。
- 快速连续切换时，迟到的旧响应不能覆盖新项目。
- 空建议和请求失败均不显示令狐冲固定示例，且仍可手动提问。
- 点击动态建议可填入并提交现有问答流程。

## 验收标准

1. 选择《书剑恩仇录》后，页面显示“向《书剑恩仇录》图谱提问”。
2. 示例人物和问题来自《书剑恩仇录》当前图谱，且逐条提交都能由现有受控模板返回有证据的答案。
3. 从任一项目切换到另一项目时，不刷新页面也不会看到旧项目名称、建议或答案。
4. 空图谱不会展示其他项目的默认示例。
5. 不新增 LLM 调用，建议接口响应保持有界且只读。
