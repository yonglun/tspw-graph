# 实体属性与图谱性能优化设计

## 1. 背景

当前系统的本体目录只定义实体类型的名称、说明、颜色和继承关系。图谱实体只保存名称、别名和描述，实体详情主要展示关系事实及原文证据。图谱页选择搜索结果后，还会等待实体详情与二度关系图同时完成，再一次性更新界面。

本次改进同时解决三个问题：

1. 为每种本体类型定义可抽取、可验证的属性。
2. 点击图谱实体后展示属性、关系摘要及各自证据。
3. 缩短从选择实体到图谱可见的等待时间。

## 2. 已确认的产品决策

- 属性采用预定义模式，由模型从原文自动抽取，并且每个属性值必须有原文证据。
- 师徒、配偶、隶属、持有等实体间语义继续建模为关系，不复制为属性。
- 属性采用独立的“属性断言”模型，而不是实体节点上的自由 JSON 字典。
- 现有项目使用原始 TXT 增量补抽属性，不删除或重建已有图谱。
- 属性第一阶段只读，展示值、来源章节和原文证据。
- 图谱默认只加载一度关系，二度关系由用户主动展开。
- 实体详情采用已批准的“身份、属性、关系、证据”分区布局。

## 3. 范围

### 3.1 本期包含

- 本体属性定义和父类型属性继承。
- 模型结构化输出、属性规范化、证据校验和拒绝统计。
- Neo4j 属性断言及证据关系的增量写入。
- 属性补抽任务及构建页入口。
- 实体详情 API 的属性与关系摘要扩展。
- 实体详情面板的属性、关系和证据展示。
- 搜索、邻居查询、连接复用、前端请求调度和布局性能优化。
- 自动测试与部署环境性能检查脚本。

### 3.2 本期不包含

- 属性人工编辑。
- 属性审核队列、版本历史或冲突合并界面。
- 用户自定义本体属性。
- 全图一次性加载。
- 将已有关系转换为属性。

## 4. 本体属性模式

### 4.1 属性定义

`EntityTypeDefinition` 增加 `property_definitions`。每个 `PropertyDefinition` 包含：

- `id`：稳定的英文编号。
- `label`：中文名称。
- `description`：供用户和抽取模型理解的定义。
- `value_type`：`TEXT`、`NUMBER`、`BOOLEAN` 或 `ENUM`。
- `multiple`：是否允许多个值。
- `enum_values`：枚举允许值；非枚举为空。

子类型继承父类型属性。目录 API 返回某类型的直接属性以及解析后的有效属性，避免前端重复实现继承算法。

### 4.2 第一版属性集合

第一版控制字段数量，避免模型因属性过多而降低抽取稳定性。

| 类型 | 属性 |
| --- | --- |
| 通用实体 | 名称、别名、描述（继续使用实体基础字段） |
| Person | `gender` 性别、`honorific` 称号（多值）、`identity` 身份（多值）、`life_status` 人物状态 |
| Organization 及子类 | `characteristic` 组织特征（多值）、`activity_region` 活动区域（多值） |
| MartialArt 及子类 | `weapon_type` 所用兵器、`characteristic` 功法特征（多值）、`prerequisite` 修炼条件（多值）、`effect` 招式效果（多值） |
| Event 及子类 | `event_kind` 事件类别、`outcome` 结果（多值） |
| Place | `place_kind` 地点类别、`region` 所属区域 |
| Artifact | `artifact_kind` 物品类别、`material` 材质、`characteristic` 特征（多值） |

涉及其他实体的值必须优先使用关系。例如“岳不群是令狐冲的师父”使用 `MASTER_OF`，不能写成 `Person.master = 岳不群`。

## 5. 属性断言数据模型

### 5.1 领域模型

新增 `AttributeAssertionRecord`：

- `id`
- `project_id`
- `entity_id`
- `property_id`
- `value`
- `value_type`
- `confidence`
- `evidence_ids`

稳定编号由 `project_id + entity_id + property_id + normalized_value` 生成。同一属性值再次抽取时复用断言节点，并补充新的证据边。

### 5.2 Neo4j 表达

```text
(:Entity)-[:HAS_ATTRIBUTE]->(:AttributeAssertion)-[:EVIDENCED_BY]->(:Evidence)
```

新增复合唯一约束：

```text
(AttributeAssertion.project_id, AttributeAssertion.id)
```

属性断言保留项目边界。删除项目时，现有按 `project_id` 删除的流程同时清除属性断言。

### 5.3 抽取与规范化

模型结果增加 `attributes`，每项包含：

- `entity_local_id`
- `property_id`
- `value`
- `evidence`
- `confidence`

规范化层负责：

- 验证实体引用存在。
- 验证属性属于实体类型的有效属性集合。
- 将值转换并校验为定义的类型。
- 拒绝空值、超长值、非法枚举和无效证据。
- 将证据偏移对齐到当前分片并转换为全文绝对位置。
- 生成稳定断言编号并去重。

属性项验证失败只产生拒绝记录，不使整个分片失败。关系事实沿用现有验证流程。

## 6. 现有项目的属性补抽

### 6.1 任务类型

`Job` 增加 `kind` 字段：

- `FULL_BUILD`：现有完整构建流程，作为旧记录的默认值。
- `ATTRIBUTE_BACKFILL`：只抽取并导入属性断言与属性证据。

SQLite 启动时执行兼容升级：若旧 `jobs` 表缺少 `kind`，增加非空字段并将历史记录回填为 `FULL_BUILD`。不要求用户删除数据卷。

### 6.2 接口

新增：

```http
POST /api/projects/{project_id}/attribute-jobs
Content-Type: application/json

{"model_profile_id":"azure:gpt-4o-mini"}
```

成功返回现有 `JobSnapshot` 扩展结构，状态码 `201`。项目不存在返回 `404`；原始 TXT 不存在返回 `409 PROJECT_SOURCE_MISSING`；模型配置不可用返回现有模型配置错误语义。

### 6.3 Worker 行为

Worker 根据 `job.kind` 路由到完整构建或属性补抽处理器。补抽任务：

1. 读取项目原始 TXT。
2. 使用相同章节和分片规则。
3. 只保留能够解析到现有实体的属性断言。
4. 不重写实体、关系、关系审核状态或实体合并结果。
5. 幂等导入断言及证据。
6. 保存属性专项质量报告。

质量报告增加：接受属性数、接受属性证据数以及按原因分类的属性拒绝数。

### 6.4 构建页

当前项目存在原始 TXT 时显示“重新抽取属性”。用户选择模型配置后创建补抽任务，并复用当前任务进度组件。原始文件不存在时按钮禁用并显示原因。

## 7. 图谱和实体详情接口

### 7.1 向后兼容

现有接口路径保持不变。响应只增加可选或有默认值的字段。

`EntityDetail` 增加：

- `attributes: AttributeDetail[]`
- `relations: RelationSummary[]`

`AttributeDetail` 包含属性编号、标签、类型、值、置信度和证据列表。`RelationSummary` 包含关系编号、中文标签、方向、对端实体摘要和事实编号。原有 `facts` 保留，用于关系原文证据和审核入口。

### 7.2 查询策略

实体详情使用 Neo4j 子查询分别聚合属性断言和关系事实，避免属性、事实与多条证据之间形成笛卡尔乘积。没有属性时返回空数组。

一度邻居使用专用查询：

```text
MATCH (center:Entity {project_id, id})
OPTIONAL MATCH (center)-[r:RELATED]-(other:Entity {project_id})
...
RETURN center, other, r
LIMIT ...
```

该查询同时证明中心实体存在，因此服务层不再额外执行 `entity_exists`。二度关系继续使用受限路径查询，并且只在用户主动请求时执行。

## 8. 前端交互

点击搜索结果后：

1. 立即用搜索结果绘制中心实体并关闭搜索结果列表。
2. 并行请求一度图谱和实体详情。
3. 一度图谱返回后立即更新画布，不等待详情。
4. 详情返回后更新右侧面板。
5. “展开二度关系”按钮请求 `depth=2` 并替换当前画布。

实体详情面板依次显示：

1. 类型、名称、描述和别名。
2. 本体属性。
3. 关系摘要。
4. 属性证据。
5. 关系证据及现有“加入审核”操作。

没有属性时显示“尚未抽取到有证据支持的属性”。缺少详情时，图谱仍然可用。

前端为搜索和实体选择请求使用 `AbortController`。项目切换、查询变化或选择新实体时取消旧请求，取消错误不显示给用户。

## 9. 性能设计

### 9.1 Neo4j 连接

Neo4j Driver 在 FastAPI 生命周期内创建一次，API 请求复用其连接池；应用退出时关闭。Repository 不再拥有每个请求独立创建和关闭 Driver 的生命周期。

### 9.2 索引与搜索

新增索引：

- `(Entity.project_id, Entity.name)`
- `(Entity.project_id, Entity.type)`

搜索先执行可命中索引的精确名称查询；结果不足时再执行名称及别名的包含查询。精确结果始终排在首位，合并结果按实体编号去重，并继续排除已合并实体。

### 9.3 图布局

一度和二度图谱使用非迭代、确定性的中心辐射布局，替换当前成本较高的 `cose` 布局。首屏限制最多 50 个节点；二度关系限制最多 100 个节点。

### 9.4 性能预算

以当前 Azure 部署环境的常规人物实体为验收对象：

- 点击后中心实体立即可见，不依赖网络请求。
- 搜索 API 服务端目标 P95 不超过 300 ms。
- 一度邻居 API 服务端目标 P95 不超过 700 ms。
- 50 个节点内的前端布局目标不超过 100 ms。
- 一度图谱从点击到可见目标不超过 1 秒。
- 详情请求不阻塞图谱展示。

自动化测试验证调用顺序和查询边界；部署环境脚本记录真实耗时，不在单元测试中伪造网络时延结论。

## 10. 错误处理

- 未定义属性、无效类型、空值、错误实体引用和证据不匹配：拒绝单条属性并计入质量报告。
- Azure 429 或临时网络错误：沿用退避重试。
- Azure 内容过滤：记录跳过分片，继续处理其余分片。
- 项目原始文件缺失：不创建补抽任务，返回结构化冲突错误。
- Neo4j 不可用：沿用 `GRAPH_UNAVAILABLE`；前端保留中心实体并显示错误，不显示半完成的邻居图。
- 请求被前端主动取消：静默结束，不覆盖当前状态。
- 二度展开失败：保留已经显示的一度图谱并提示用户重试。

## 12. 发布与回滚

本次变更为加法式升级。旧实体和事实 API 字段继续保留；没有属性断言的项目仍可正常浏览。回滚应用版本不会删除新增 Neo4j 节点或 SQLite `kind` 字段，旧版本会忽略这些数据。部署前备份 SQLite 和 Neo4j 数据卷，部署后先对一个小项目执行属性补抽，再对“笑傲江湖完整版”运行完整补抽。
