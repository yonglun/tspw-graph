# 管理员认证与受保护管理模块设计

日期：2026-07-14

状态：已确认设计，待实施计划

## 背景与目标

系统当前没有管理员身份体系。“构建”和“审核”功能与公开浏览功能使用同一套导航与接口，任何访问者都能看到管理入口并发起改变数据的操作。系统也缺少管理员账号维护、密码重置、会话撤销和操作审计能力。

本次改造目标：

1. 增加管理员登录、退出、修改密码和会话管理。
2. 增加管理员管理模块，用于创建、编辑、启用、停用和重置管理员密码。
3. 只有已登录且已完成强制改密的管理员才能访问“管理员”“构建”“审核”三个模块。
4. 未登录时不显示上述三个导航入口，也不显示图谱中的“加入审核”操作。
5. 后端对所有受保护接口执行真实鉴权，不能只依赖前端隐藏入口。
6. 首次启动提供缺省管理员 `admin` / `Pass@word1`，首次登录必须修改密码。
7. 记录管理员安全操作审计，同时不记录密码、会话令牌等敏感信息。

## 非目标

- 不增加普通用户注册、注册审批或找回密码邮件。
- 不设计多角色、细粒度权限或项目级授权；所有管理员权限相同。
- 不把公开的导览、本体、图谱浏览、故事线和问答改成登录后可用。
- 不引入第三方身份提供商、OAuth、OIDC、LDAP 或单点登录。
- 不永久删除管理员账号；账号生命周期仅使用启用和停用状态。
- 不改变 Worker 的任务消费协议，也不让 Worker 持有浏览器会话。

## 方案选择

采用“服务端不透明会话 + HttpOnly Cookie”方案。

登录成功后，API 生成高熵随机会话令牌，通过 Cookie 发送给浏览器；数据库只保存令牌摘要。浏览器不在 `localStorage` 或 `sessionStorage` 中保存密码、令牌或其他登录凭据。服务端会话便于即时撤销、强制改密、账号停用和审计，也适合当前 FastAPI、React、SQLite 和同源 Nginx 部署结构。

未采用的方案：

- JWT：虽然便于无状态扩展，但即时撤销、强制改密和账号停用仍需服务端状态，增加刷新令牌与黑名单复杂度。
- Nginx Basic Auth：无法提供应用内管理员维护、强制改密、细致审计和友好的登录体验。

## 总体架构

### 后端模块边界

新增独立 `auth` 模块，职责如下：

- 管理员账号仓储：账号查询、唯一性、启用状态和密码摘要。
- 密码服务：Argon2id 哈希、验证和密码策略检查。
- 会话服务：创建、校验、滚动过期、撤销和清理会话。
- 登录限流服务：记录失败次数和临时锁定状态。
- 审计服务：记录安全相关管理操作。
- FastAPI 鉴权依赖：向受保护路由提供当前管理员，并统一生成认证错误。
- CSRF 校验依赖：保护所有使用 Cookie 认证的状态变更请求。

现有项目、任务和审核模块继续负责各自业务。它们通过鉴权依赖声明权限边界，不直接实现密码或会话逻辑。

### 前端模块边界

新增认证状态层，职责如下：

- 应用启动时调用会话接口恢复登录状态。
- 提供当前管理员、是否必须改密、CSRF 值和加载状态。
- 处理登录、退出、修改密码及会话失效。
- 控制受保护导航、路由和操作入口的可见性。
- 使用 `BroadcastChannel` 同步同一浏览器多个标签页的登录、退出和会话失效状态。

管理员页面只负责账号列表和账号操作，不自行判断权限规则；最终规则由后端保证。

## 数据模型

所有新表存放在现有 SQLite 数据库中，由当前 SQLAlchemy `create_all` 初始化机制创建。

### `admin_accounts`

至少包含：

- `id`：稳定的管理员标识。
- `username`：显示用户名。
- `normalized_username`：用于不区分大小写的唯一查询。
- `password_hash`：Argon2id 密码摘要。
- `is_enabled`：是否允许登录。
- `must_change_password`：是否必须在继续使用管理功能前修改密码。
- `created_at`、`updated_at`。
- `password_changed_at`。

用户名规范：去除首尾空白后长度为 3–64 个字符，并以规范化值建立唯一约束。修改用户名不得绕过唯一约束。

### `admin_sessions`

至少包含：

- `id`。
- `admin_id`。
- `token_hash`：随机会话令牌的摘要，不保存原始令牌。
- `csrf_hash`：CSRF 值的摘要。
- `created_at`、`last_seen_at`、`expires_at`。
- `ip_address`、`user_agent`：在会话创建时记录；上游未提供时允许为空，仅用于安全审计和会话识别。

会话采用 8 小时闲置超时。有效请求更新 `last_seen_at` 和 `expires_at`。Cookie 使用持久化 `Max-Age`，因此关闭并重新打开浏览器后，只要会话未过期或撤销仍可恢复。

### `admin_login_throttles`

至少包含规范化用户名、来源 IP、失败次数、最近失败时间和 `locked_until`。该表只服务于临时登录锁定，不作为账号是否存在的外部信号。

### `admin_audit_events`

至少包含：

- `id`、`created_at`。
- `actor_admin_id` 和当时的操作者用户名快照。
- `target_admin_id` 和目标用户名快照（如适用）。
- `action`、`result`。
- `ip_address`。
- 不含敏感值的结构化元数据。

审计日志不得写入明文密码、密码摘要、Cookie、原始会话令牌或 CSRF 值。

## 缺省管理员初始化

API 启动时执行幂等初始化：

1. 仅当 `admin_accounts` 表完全为空时创建缺省管理员。
2. 缺省用户名为 `admin`，缺省初始密码为 `Pass@word1`。
3. 密码立即以 Argon2id 保存，绝不以明文写入数据库或日志。
4. 新账号设置为启用，且 `must_change_password = true`。
5. 数据库已经存在任何管理员时，不创建、不覆盖、也不重置账号。

部署可通过环境变量覆盖首次初始化的用户名和密码；若未配置则使用上述缺省值。生产部署文档必须明确提示首次登录立即改密。

## 会话与 Cookie

- 会话令牌使用密码学安全随机数，Cookie 名称固定且不携带账号信息。
- Cookie 设置 `HttpOnly`、`SameSite=Lax` 和明确的 `Path=/`。
- HTTPS 环境设置 `Secure`；本地 HTTP 开发环境可通过显式配置关闭。
- 登录和有效会话恢复接口向前端返回当前管理员、`must_change_password` 和本次会话的 CSRF 值。
- CSRF 值只保存在前端内存中，状态变更请求通过专用请求头发送；后端与会话记录中的摘要比较。
- 退出登录撤销当前会话并清除 Cookie。
- 管理员停用或密码被其他管理员重置时，撤销该账号全部会话。
- 管理员修改自己的密码时，撤销该账号其他会话；当前改密会话可以继续，并清除强制改密标记。
- 过期会话由请求路径惰性删除，并由周期性清理任务批量清理。

## 密码与登录安全策略

密码必须：

- 至少 10 个字符；
- 至少包含一个大写字母；
- 至少包含一个小写字母；
- 至少包含一个数字；
- 至少包含一个特殊字符。

登录失败策略：

- 同一“规范化用户名 + 来源 IP”连续失败 5 次后锁定 15 分钟。
- 用户名不存在和密码错误使用相同的外部错误信息与近似处理路径，避免账号枚举。
- 锁定响应可以显示剩余等待时间，但不能确认账号是否存在。
- 登录成功后清除该组合的失败计数。
- 已停用账号不能登录，且已有会话立即失效。

## 管理员生命周期规则

所有管理员权限相同，可以：

- 创建管理员；
- 修改管理员用户名；
- 启用或停用其他管理员；
- 为其他管理员设置符合策略的临时密码；
- 查看管理员安全审计记录。

约束：

- 不支持永久删除管理员。
- 管理员不能停用当前登录的自己。
- 系统中必须至少保留一个启用的管理员，不能停用最后一个有效管理员。
- 重置密码后，目标管理员设置为 `must_change_password = true`，其全部会话立即撤销。
- 创建管理员时由操作者设置临时密码；新管理员首次登录必须修改密码。
- 修改用户名、启停账号和重置密码必须在数据库事务中执行，并由数据库唯一约束处理并发冲突。

系统不提供匿名“忘记密码”入口。正常恢复由另一个已登录管理员执行密码重置。如果没有任何管理员能够登录，部署人员可在 API 容器内运行受控的一次性恢复命令，为指定的现有管理员设置临时密码。该命令不创建额外账号，不绕过密码策略；它会设置 `must_change_password = true`、撤销目标全部会话并以系统恢复事件写入审计日志。

## API 设计

### 认证接口

```text
POST /api/auth/login
GET  /api/auth/session
POST /api/auth/logout
POST /api/auth/change-password
```

`login` 接收用户名和密码；成功后设置会话 Cookie。`session` 用于应用启动和页面刷新时恢复会话。`change-password` 同时支持首次强制改密和已登录管理员主动改密。

### 管理员接口

```text
GET   /api/admins
POST  /api/admins
PATCH /api/admins/{admin_id}
POST  /api/admins/{admin_id}/enable
POST  /api/admins/{admin_id}/disable
POST  /api/admins/{admin_id}/reset-password
GET   /api/admin-audit-events
```

所有管理员接口都要求有效会话、已完成强制改密，并对状态变更请求校验 CSRF。

### 受保护业务接口

下列操作要求有效管理员会话和已完成强制改密：

- 构建页面使用的项目上传、属性重抽取和任务创建操作。
- 构建任务的查询、暂停、继续、取消、失败片段重试等控制接口。
- 构建页面使用的模型配置查询。
- 所有审核队列、审核详情、接受、拒绝、忽略和实体合并接口。
- 图谱“加入审核”创建审核项的接口。

公开能力保持匿名可用：

- 导览、本体、图谱浏览、故事线和问答。
- 项目列表及公开页面所需的项目只读元数据。
- 图谱搜索、实体详情、关系与属性证据的只读查询。

即使前端没有显示入口，后端受保护接口也必须独立执行鉴权。

## 统一错误语义

- `401 AUTHENTICATION_REQUIRED`：没有会话、会话过期或会话已撤销。
- `403 PASSWORD_CHANGE_REQUIRED`：会话有效，但必须先修改密码。
- `403 CSRF_VALIDATION_FAILED`：状态变更请求缺少或携带无效 CSRF 值。
- `403 ACCOUNT_DISABLED`：账号已停用。
- `423 ACCOUNT_LOCKED`：登录尝试被临时锁定；响应包含可显示的剩余秒数。
- `409 CANNOT_DISABLE_SELF`：试图停用当前账号。
- `409 LAST_ACTIVE_ADMIN`：试图停用最后一个启用的管理员。
- `409 USERNAME_ALREADY_EXISTS`：用户名规范化后冲突。
- `422 PASSWORD_POLICY_VIOLATION`：密码不符合策略；响应列出未满足的规则。

登录凭据错误统一使用不暴露用户名是否存在的提示。前端使用错误码而不是解析中文消息决定跳转和界面状态。

## 前端导航与页面流程

### 未登录状态

- 顶部导航只显示：导览、本体、图谱、故事线、问答。
- 顶部右侧显示“管理员登录”。
- 不显示：管理员、构建、审核。
- 图谱详情不显示“加入审核”。
- 直接访问 `/admin`、`/build` 或 `/review` 时跳转到登录页，并保存原始地址为 `returnTo`。

### 登录与强制改密

1. 管理员提交用户名和密码。
2. 如果账号要求改密，立即进入强制修改密码页，不能进入其他受保护页面。
3. 修改成功后，优先返回合法的同源 `returnTo`；没有原地址时进入管理员页面。
4. 普通登录成功后同样返回原访问页。
5. 登录失败显示统一提示；锁定时可显示剩余等待时间，但不暴露账号存在性。

`returnTo` 仅允许应用内相对路径，禁止外部 URL，避免开放重定向。

### 已登录状态

- 顶部导航增加：管理员、构建、审核。
- 右上角“管理员登录”替换为账号菜单。
- 账号菜单提供“修改密码”和“退出登录”。
- 图谱详情恢复“加入审核”。

管理员页面采用已确认布局：

- 主区域显示管理员列表、状态、是否需要改密及最近更新时间。
- 创建、编辑、重置密码、启用和停用通过聚焦的弹窗或抽屉完成。
- 停用和密码重置需要二次确认，并明确说明会话撤销影响。
- 当前账号和最后一个启用管理员的停用操作在界面禁用，并展示原因。
- 审计记录以紧凑表格展示操作者、目标、动作、结果和时间。

### 会话失效

- API 返回 `401` 时清空前端认证状态，跳转登录并提示“会话已过期”，同时保留原地址。
- API 返回 `403 PASSWORD_CHANGE_REQUIRED` 时进入强制改密页。
- API 返回 `403 CSRF_VALIDATION_FAILED` 时提示刷新页面后重试，不自动重复状态变更请求。
- `BroadcastChannel` 通知其他标签页同步注销或重新获取会话。
- 页面刷新通过 `/api/auth/session` 恢复状态，不从浏览器存储读取凭据。

## 数据流

### 登录

1. 前端向登录接口提交凭据。
2. 后端规范化用户名，检查组合锁定状态并以恒定语义验证密码。
3. 验证成功后创建服务端会话、设置 Cookie、生成 CSRF 值并写入登录审计。
4. 前端根据 `must_change_password` 进入强制改密或目标页面。

### 受保护请求

1. 浏览器自动携带 HttpOnly 会话 Cookie。
2. 鉴权依赖对令牌取摘要并查询会话，校验管理员启用状态和闲置过期时间。
3. 状态变更请求额外校验 CSRF 请求头。
4. 通过后更新会话活动时间，再由业务路由处理请求。

### 密码重置或账号停用

1. 当前管理员提交操作并完成二次确认。
2. 后端在事务中检查不能停用自己、不能停用最后一个启用管理员等约束。
3. 更新目标账号并撤销目标全部会话。
4. 写入成功或失败审计事件。
5. 目标管理员的下一次请求返回 `401`；重置密码后重新登录会进入强制改密。

### 构建任务

API 在创建或控制构建任务前完成鉴权。任务进入现有持久化队列后，Worker 按原有协议继续处理，不传递浏览器 Cookie、CSRF 值或管理员密码。

## 并发与一致性

- 管理员用户名唯一性由数据库唯一约束保证，接口在冲突时返回稳定的 `409` 错误。
- “最后一个启用管理员”检查和状态更新在同一事务内完成。
- 停用、重置密码和会话撤销在同一事务边界内完成。
- 登录限流更新使用数据库事务，避免并发失败请求丢失计数。
- 缺省管理员初始化幂等，多个 API 进程并发启动时依赖唯一约束保证最多创建一次。
- 前端切换页面或标签页同步不改变后端权威状态。

## 部署与配置

新增配置至少包括：

- 首次初始化管理员用户名和密码。
- Cookie `Secure` 开关。
- 会话闲置超时，缺省为 8 小时。
- 登录最大失败次数，缺省为 5 次。
- 登录锁定时长，缺省为 15 分钟。

Docker 部署手册需要补充：

- 首次管理员初始化和首次改密步骤。
- HTTPS 反向代理下的 Secure Cookie 配置。
- 管理员忘记密码时的受控恢复流程。
- 账号锁定的等待和诊断方法。
- 环境变量不得提交到 Git 或输出到日志。

回滚应用代码不会自动删除管理员、会话或审计表。旧版本无法识别这些表，但不会影响现有项目和图谱数据。

## 测试策略

### 后端单元与集成测试

- 空数据库只创建一次缺省管理员，重启不覆盖已有密码。
- Argon2id 哈希和密码验证正确，数据库不含明文密码。
- 密码策略逐项生效，缺省初始密码符合策略。
- 用户名查询和唯一性不区分大小写。
- 登录成功、失败计数、5 次锁定、15 分钟解锁和成功后清零正确。
- 会话 Cookie 属性、8 小时滚动闲置过期、注销和撤销正确。
- CSRF 对所有状态变更接口生效，只读接口不要求 CSRF。
- 强制改密期间受保护业务接口返回 `PASSWORD_CHANGE_REQUIRED`。
- 不能停用自己，不能停用最后一个启用管理员。
- 创建账号、重置密码和停用账号会设置正确状态并撤销相应会话。
- 审计事件覆盖成功与失败操作，且不包含敏感值。
- 所有构建、审核和管理员接口在匿名访问时拒绝请求。
- 公开图谱、本体、故事线、问答和项目只读接口继续匿名可用。

### 前端测试

- 匿名、强制改密和正常登录三种导航状态正确。
- 受保护路由跳转登录并保留安全的 `returnTo`。
- 登录后返回原页面，外部 `returnTo` 被拒绝。
- 强制改密完成前不能进入管理员、构建和审核页面。

## 验收标准

1. 全新部署可使用 `admin` / `Pass@word1` 登录，但必须修改密码后才能继续。
2. 未登录时，界面完全不显示管理员、构建、审核和加入审核入口。
3. 未登录用户即使直接调用受保护接口，也会收到稳定的认证错误。
4. 登录管理员可以维护其他管理员，但不能停用自己或最后一个启用管理员。
5. 新建账号和密码重置都要求目标管理员下次登录强制改密。
6. 密码修改、重置和账号停用会按设计撤销会话。
7. 8 小时闲置过期、登录锁定、CSRF 和审计机制均通过自动化测试。
8. 公开导览、本体、图谱、故事线和问答保持匿名可用且行为不回归。
9. 构建与审核现有业务流程在管理员登录后保持可用，Worker 无需接触登录凭据。
