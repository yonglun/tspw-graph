# 系统架构与数据库设计文档拆分规格

日期：2026-07-16

状态：已实现

## 1. 背景

现有 `docs/database-design.md` 同时描述系统组件协作、构建与审核流程、跨库一致性、Docker 运维，以及 SQLite 和 Neo4j 的具体数据模型。内容事实完整，但系统架构职责与数据库结构职责混在同一文件中，不利于按角色阅读和后续维护。

## 2. 目标

将现有内容拆分为两份能够独立阅读、互相引用且不重复大段正文的文档：

- `docs/system-architecture-design.md`：回答系统由什么组成、组件如何协作、数据如何流转以及如何运行维护。
- `docs/database-design.md`：回答数据如何组织、约束、查询、隔离和演进。

拆分不得丢失现有表结构、图模型、示例查询、风险或代码链接。

## 3. 系统架构设计文档范围

`docs/system-architecture-design.md` 包含：

1. 文档目标、适用读者和相关文档链接。
2. API、Worker、SQLite、Neo4j、上传文件和 Web 前端的职责边界。
3. 小说文本进入知识图谱的端到端数据流。
4. 完整构建、属性补抽、任务暂停/失败/重试的生命周期。
5. 项目隔离、事实审核、实体合并、别名拆分和项目删除流程。
6. SQLite 与 Neo4j 之间的跨存储一致性边界和补偿责任。
7. Docker 数据卷、备份恢复、安全设计和监控指标。
8. 系统级已知限制与演进建议。

系统架构文档只保留理解架构所需的数据模型摘要，具体字段、约束和查询链接到数据库设计文档。

## 4. 数据库设计文档范围

`docs/database-design.md` 包含：

1. 文档目标、适用读者和系统架构文档链接。
2. SQLite 与 Neo4j 的存储职责摘要。
3. SQLite ER 图、初始化方式、外键执行注意事项和 11 张表的完整字段字典。
4. Neo4j 属性图、6 类节点、8 类关系、唯一约束和索引。
5. SQLite 与 Neo4j 的业务 ID 和状态映射。
6. 数据库级项目隔离、约束、索引和性能设计。
7. 典型 SQL、Cypher 和一致性巡检查询。
8. 数据库级已知限制与演进建议。

数据库设计文档不再承担完整业务流程、容器运维或认证流程的说明；相关内容链接到系统架构文档。

## 5. 内容迁移规则

| 原文内容 | 目标文档 |
| --- | --- |
| 总体架构、组件职责、默认部署位置 | 系统架构设计 |
| 小说文本进入图谱的数据流 | 系统架构设计 |
| SQLite ER、初始化、表结构 | 数据库设计 |
| Neo4j 模型、节点、关系、约束、索引 | 数据库设计 |
| SQLite 与 Neo4j 数据映射 | 数据库设计 |
| 项目隔离和审核一致性 | 系统架构设计；数据库文档保留隔离规则摘要 |
| 索引与查询性能 | 数据库设计；系统文档保留监控指标 |
| 构建、属性补抽、任务和删除生命周期 | 系统架构设计 |
| Docker 数据卷、备份与恢复 | 系统架构设计 |
| 管理员凭据、会话、CSRF、审计和网络暴露 | 系统架构设计 |
| SQL、Cypher 和排查查询 | 数据库设计 |
| 已知限制 | 按系统级与数据库级影响分别归档 |
| 代码定位 | 两份文档分别保留与自身内容直接相关的链接 |

## 6. 链接与命名

- 两个文件放在 `docs/` 根目录。
- 两份文档的开头均增加“相关文档”段落并互相链接。
- 使用稳定的相对链接：`system-architecture-design.md` 与 `database-design.md`。
- 原有 `database-design.md` 路径继续存在，避免既有外部链接失效。

## 7. 验收标准

1. 两份文档均存在且 Markdown 代码围栏闭合。
2. 系统架构文档包含架构图、数据流、生命周期、一致性、运维和安全章节。
3. 数据库文档仍完整覆盖 11 张 SQLite 表、6 类 Neo4j 节点和 8 类关系。
4. 原有 SQL、Cypher、备份命令、已知限制和代码定位均能在其中一份文档找到。
5. 两份文档互相链接，所有仓库内链接均有效。
6. 文档不包含真实密钥、会话令牌或明文管理员密码。
7. `git diff --check` 通过。

## 8. 非目标

- 不修改应用代码、数据库 schema、Docker Compose 或运行时行为。
- 不新增第三份总览文档。
- 不借拆分机会重新定义系统能力或掩盖现有实现限制。
